# FinChronicle v4.1.0 Plan

> Phase 3: Product Value — forward-looking intelligence on top of the solid v4.0 accounting model.

**Prerequisite:** v4.0.0 shipped (✅). Account linking, reconciliation, category hierarchy, and asset/liability classification are all in place.

---

## What v4.1 delivers

Two features that use the data already in the app to answer the questions users actually ask:

1. **Cash-Flow Forecast (3.1)** — "What will my account balances look like next month?"
2. **Financial Health Alerts (3.2)** — "Am I falling behind? Is something about to go wrong?"

These are read-only. No new DB stores, no schema changes, no DB_VERSION bump.

## Feature 3.1 — Cash-Flow Forecast

### Problem
The app shows the past. A user with ₹50,000 in checking and ₹30,000 in rent due on the 1st has no warning. Recurring templates already encode the future — we just don't surface it.

### Data available (no new collection needed)
- `state.recurringTemplates` — each template has `amount`, `frequency`, `nextDueDate`, `enabled`, `type` (income/expense), `fromAccount`, `toAccount`
- `state.accounts` — current opening balances
- `getAccountBalance(name)` in `accounts.js` — current running balance per account
- `computeNextDueDate(frequency, dayOfMonth, fromDateStr)` in `recurring.js` — already computes future dates

### Design

**New module:** `js/forecast.js`

**Core function (pure, testable):**
```javascript
export function buildForecast(accounts, transactions, recurringTemplates, horizonDays = 90)
// Returns: { accountForecasts: { [accountName]: [{ date, label, amount, runningBalance }] }, warnings: [] }
```

Logic:
1. For each enabled recurring template, generate all occurrences within `horizonDays` using `computeNextDueDate`
2. Start each account from `getAccountBalance(name)` (current balance)
3. Walk events in date order, applying income/expense to linked account
4. Flag any account that goes negative → `warnings` array
5. Return timeline per account + warnings

**UI:** New section inside the Reports tab (not a new tab — keeps nav clean).

```
┌─────────────────────────────────────────────────┐
│ Cash-Flow Forecast          [30d] [60d] [90d]   │
│                                                  │
│ Checking                                         │
│ Today  ₹48,200  ──────────────────────────────  │
│ Jun 1  Rent -₹25,000        ₹23,200   ⚠ low    │
│ Jun 5  Salary +₹65,000      ₹88,200             │
│ Jun 15 Internet -₹1,200     ₹87,000             │
│                                                  │
│ ⚠ Checking goes below ₹5,000 on Jun 1           │
└─────────────────────────────────────────────────┘
```

**Files:**
- New `js/forecast.js`
- `js/app.js` — import, bind horizon toggle buttons
- `index.html` — forecast section in Reports tab
- `css/styles.css` — forecast timeline styles
- `sw.js` — add `./js/forecast.js` to CACHE_URLS

**Risk:** Low. Read-only. Only shows if user has recurring templates with account links. Degrades gracefully (shows nothing) if no data.

**Constraint:** Only accounts with at least one linked recurring template get a forecast row. Unlinked accounts are skipped — no phantom projections.

## Feature 3.2 — Financial Health Alerts

### Problem
Current alerts in `js/alerts.js` detect historical patterns (spike, unusual amount, velocity, drift). They fire *after* the fact. Users need forward-looking and habit-forming alerts.

### Four new alert types (additive to existing system)

#### 3.2.1 Inactivity Alert
**Trigger:** No transaction logged in 5 days (configurable; default 5).
**Message:** "No transactions logged in 5 days — are you up to date?"
**Why:** App abandonment is the #1 failure mode of personal finance tools. This fights it.
**Implementation:** Check `state.transactions` sorted by `createdAt`; compare latest to today.
**Storage:** Last-seen date in `localStorage` key `last_transaction_date` (already computable from state).

#### 3.2.2 Bill Due Alert
**Trigger:** A recurring expense is due within N days (default 3) and the linked account balance is below the bill amount × 1.2.
**Message:** "Rent (₹25,000) is due in 2 days — Checking balance is ₹18,000."
**Why:** The most actionable alert possible for a finance app.
**Implementation:** Walk `state.recurringTemplates` where `type === 'expense'` and `nextDueDate` is within N days. Check `getAccountBalance(template.fromAccount)` against `template.amount`. Only fires if account is linked.

#### 3.2.3 Savings Rate Trend Alert
**Trigger:** Savings rate below threshold (default 10%) for 3 consecutive months.
**Message:** "Savings rate has been below 10% for 3 months (avg 4.2%)."
**Why:** Single-month dip is noise. Sustained trend is signal.
**Implementation:** Use `getSavingsRate(month)` from `savings.js` for the last 3 months. If all three are below threshold, fire. One alert per quarter (use dismiss timestamp).

#### 3.2.4 Monthly Pace Alert
**Trigger:** Current-month expense pace projects to exceed the highest active budget by > 20%.
**Message:** "Food spending is on pace for ₹8,400 this month — budget is ₹6,000."
**Why:** Budget warnings currently fire only *after* the limit is hit. This warns while there's still time.
**Implementation:** For each budget category, calculate `(spending so far / days elapsed) × days in month`. If projected > `budget.limit × 1.2`, fire alert.

### Integration with existing alert system

All four types use the **same storage and render path** as current alerts:
- `runAlertChecks()` extended with `runHealthAlertChecks()`
- Same `localStorage` key `smartAlerts` (append to existing structure)
- Same `dismissAlert(id)` / `renderAlertBanners()` flow
- New alert `type` values: `'inactivity'`, `'bill-due'`, `'savings-rate-trend'`, `'monthly-pace'`

**Files:**
- `js/alerts.js` — add `runHealthAlertChecks()`, extend alert type handling
- No new files, no DB changes

**Risk:** Low. Purely additive. All four checks read existing state — no new data collection.

## Test plan

### Unit tests (pure functions)

`buildForecast()` is fully unit-testable (pure, no DOM/DB):

| Test | Assertion |
|---|---|
| No recurring templates → empty forecast | `accountForecasts === {}` |
| Single monthly expense → one event at nextDueDate | event date matches |
| Expense takes balance negative → warning fired | `warnings.length > 0` |
| 90-day horizon generates correct count for weekly template | 13 events |
| Only enabled templates included | disabled template omitted |
| Income template credited to toAccount | balance increases |

Health alert checks (inactivity, pace) are also pure given mocked state — unit-testable.

### E2E tests (Playwright)

- Alert banners appear in the UI after trigger conditions are met
- Dismiss clears the banner
- Forecast section renders when recurring templates with account links exist
- Horizon toggle (30/60/90d) changes the event count
